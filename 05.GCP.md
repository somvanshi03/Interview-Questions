### üó∫Ô∏è GCP Resource Hierarchy

To effectively organize and govern these resources, it's essential to understand GCP's hierarchical structure :

- **Organization**: The root node in the hierarchy, representing your company. All resources and policies are ultimately governed here.
- **Folders**: Containers within an organization used to group projects based on departments, teams, or applications. Policies applied at folder level are inherited by all contained projects.
- **Projects**: The fundamental level for organizing and managing GCP resources. Each project has separate settings, permissions, and billing. Every resource belongs to exactly one project.
- **Resources**: The individual services themselves (Compute Engine instances, Cloud Storage buckets, etc.).

This hierarchy enables administrators to apply IAM policies and organization rules consistently across teams and environments .

### üí° How to Use This Information

As an architect, you will combine these resources to build solutions. For example, to host a scalable web application, you might use **Cloud Run** or **GKE** for compute, configure a **VPC** for private networking, use **Cloud Load Balancing** for traffic distribution, store data in **Cloud SQL** and static files in **Cloud Storage**, and manage everything with **Deployment Manager** templates .
